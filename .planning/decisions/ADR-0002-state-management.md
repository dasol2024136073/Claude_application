# ADR-0002 — 상태관리: Riverpod

- **날짜**: 2026-05-12
- **상태**: 확정

---

## 배경

앱 내에서 관리해야 하는 상태가 복잡하다.

- 인증 상태 (로그인 여부, 사용자 토큰)
- AI 경로 생성 결과 (비동기 로딩 → 결과 → 오류)
- 실시간 날씨 데이터 (주기적 갱신)
- 커뮤니티 피드 (페이지네이션)

여러 화면이 같은 상태를 공유해야 하고, 비동기 처리가 많아 상태관리 도구 없이는 코드 복잡도가 급격히 올라간다.

---

## 결정

**Riverpod 2.x** (`flutter_riverpod`)를 선택한다.

---

## 대안

| 대안 | 검토 이유 | 제외 이유 |
|------|-----------|-----------|
| Provider | 가장 많이 쓰임, 공식 추천 | Riverpod의 구 버전, 현재는 Riverpod이 후속 |
| BLoC / Cubit | 엄격한 단방향 흐름, 대규모 팀에 적합 | 보일러플레이트 코드 과다, 혼자 개발에 부담 |
| GetX | 가장 간단한 API | 과도한 추상화, 테스트 어려움, 커뮤니티에서 안티패턴 지적 많음 |
| setState 만 사용 | 학습 비용 없음 | 화면 간 상태 공유 불가, 비동기 로딩 처리 난해 |

---

## 결과

- **장점**: `AsyncValue`로 로딩·오류·데이터 상태를 타입 안전하게 처리, 컴파일 타임 오류 잡기, Provider보다 DI(의존성 주입) 구조 명확
- **단점**: Provider보다 초기 학습 곡선이 있음
- **영향**: `lib/core/providers/` 폴더 구조로 도메인별 Provider 분리 → ADR-0001 Flutter 선택의 연장선

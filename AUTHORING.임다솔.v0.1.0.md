# AUTHORING.임다솔.v0.1.0.md

> 이 문서는 임다솔이 본 프로젝트에서 AI Agent(Claude Code)를 활용한 방식을 기록한 개인 방법론 문서입니다.
> 발표 시 "어떻게 AI를 도구로 썼는가"를 설명하는 근거 자료로 사용합니다.

---

## 1. 개요

### AI Agent로 무엇을 했는가

기획 문서 작성부터 코드 생성·리뷰까지 전 과정에 Claude Code(claude-sonnet-4-6)를 AI Agent로 활용했다.
단순히 "질문하고 복붙"하는 방식이 아니라, **Agent에게 맥락을 유지시키고 산출물을 직접 파일로 생성**하게 하는 방식을 택했다.

### 핵심 원칙

1. AI가 만든 산출물은 반드시 내가 읽고 이해한 후 commit한다.
2. 모호한 결정은 AI가 선택지를 제시하고, 최종 결정은 내가 한다.
3. AI와의 대화 흐름(프롬프트 패턴)을 `notes/` 폴더에 기록해 재사용한다.

## 2. Agent 구성

### 사용 도구

| 도구 | 역할 |
|------|------|
| Claude Code (CLI) | 파일 생성·편집, git 조작, 코드 생성 전 과정 |
| 모델 | claude-sonnet-4-6 |
| 실행 환경 | Windows 11, PowerShell / Git Bash |

### Agent에게 부여한 컨텍스트

매 세션 시작 시 Agent는 다음을 자동으로 로드한다.

- `CLAUDE.md` (프로젝트 루트) — 프로젝트 목표, 기술 스택, 금지 사항
- `.planning/*.md` — 기획 문서 전체
- `notes/` — 이전 세션의 LLM Wiki (누적 학습)

이 구조 덕분에 "지난 주에 정한 거 기억해?" 라고 묻지 않아도 Agent가 맥락을 유지한다.

## 3. Skills (slash commands / 반복 작업 자동화)

프로젝트에서 반복적으로 쓴 AI Agent 호출 패턴을 정리한 것.

### /plan — 산출물 먼저 확인 후 실행

```
[작업 설명]. 시작 전에 만들 파일 목록과 각 파일의 핵심 내용을 먼저 보여줘.
내가 OK하면 그때 만들어줘.
```

**왜 씀**: AI가 바로 파일을 만들면 내가 이해 못한 채로 commit하게 됨.
계획을 먼저 보면 "이 부분은 다르게 해줘"라고 수정 지시를 내릴 수 있음.

---

### /explain — 만든 것 설명 요청

```
방금 만든 [파일명]에서 [개념]이 어떻게 동작하는지
비전공자에게 설명하듯 한 단락으로 설명해줘.
```

**왜 씀**: 발표 때 "이게 뭔가요?" 질문에 답하려면 내가 직접 이해해야 함.
AI가 만들어줬어도 설명을 들어야 진짜 내 것이 됨.

---

### /compare — 기술 선택 비교표 요청

```
[기술 A] vs [기술 B]를 이 프로젝트 조건(Flutter, 6주, 1인 개발)에 맞춰
표로 비교해줘. 추천도 한 줄로 해줘.
```

**왜 씀**: ADR 작성 시 대안을 객관적으로 정리하기 위해.
내가 몰라서 못 쓰는 게 아니라 "비교해봤고 이게 더 나아서 선택"임을 증명.

---

### /wiki — LLM Wiki 항목 추가

```
오늘 [주제]에 대해 배운 것을 notes/week{N}-llm-wiki.md에
"## [주제]" 섹션으로 추가해줘. 형식은 기존 항목 스타일 유지.
```

**왜 씀**: 매주 새로 배운 프롬프트 패턴과 실패 사례를 누적 기록하기 위해.
다음 주 Agent가 이 파일을 읽고 같은 실수를 반복하지 않음.

## 4. Rules (AI와 협업할 때 내가 정한 규칙)

### R-01. AI 출력은 반드시 읽고 commit한다

AI가 파일을 만들면 `git diff`로 확인 후 commit.
"AI가 만들었으니 OK겠지"로 넘기지 않음.

### R-02. 모호한 질문은 먼저 선택지로 좁힌다

"백엔드 뭐 쓸까?" → "Firebase vs Supabase, 이 프로젝트 조건에서 비교해줘"
질문이 구체적일수록 AI 답변의 품질이 올라감.

### R-03. 실패한 프롬프트도 기록한다

잘 안 된 프롬프트 패턴도 `notes/`에 기록.
"이렇게 물으면 안 된다"는 것도 암묵지.

### R-04. 한 세션에 한 가지만 요청한다

"기획 문서 만들고, 코드도 짜고, 테스트도 해줘"는 실패함.
하나의 명확한 목표로 세션을 나눔.

### R-05. AI가 모른다고 하면 범위를 좁혀 다시 묻는다

"이 에러 고쳐줘" → "이 에러에서 [특정 부분]만 설명해줘"
에러 전체를 던지면 관련 없는 곳을 바꾸기도 함.

## 5. Commands (실제 사용한 CLI 명령어 모음)

### 프로젝트 초기 세팅

```bash
# Claude Code 시작
claude

# 파일 변경 사항 확인 후 commit
git add .planning/
git commit -m "Add planning docs"
git push origin main
```

### 문서 생성 패턴

```
# 세션 시작 시
"이어서 할게" → Agent가 MEMORY.md에서 이전 컨텍스트 자동 로드

# 산출물 생성 요청
".planning/decisions/ 폴더에 ADR-0001 만들어줘.
배경/결정/대안/결과 구조로, 우리가 Flutter 선택한 이유 담아서."
```

### 주차별 진행 확인

```
"week-{N}.pdf 분석해서 아직 못 끝낸 항목 알려줘"
→ PDF를 직접 읽고 현재 리포 상태와 대조해서 미완성 목록 출력
```

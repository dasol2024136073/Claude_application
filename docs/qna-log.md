## 3. 중간 → 기말: 달라진 구현 사항

중간 발표 때 "예정"이라고 말한 것들이 기말까지 실제로 어떻게 구현됐는지, 그리고 **계획과 달라진 부분**.

| 항목 | 중간 발표 때 말한 것 | 실제 기말 구현 | 이유 |
|---|---|---|---|
| AI 엔진 | Claude API | **Gemini 2.5 Flash** | 수업 슬라이드에 Claude라 적었지만 Gemini가 무료 티어·Flutter 공식 예제 더 풍부 |
| 상태관리 | Riverpod 2.x | **StatefulWidget** | Riverpod 학습 비용 > 이점. 1인 프로젝트 규모에서 오버엔지니어링 |
| 백엔드 | Firebase (Auth + Firestore) | **SharedPreferences (로컬)** | 데모 목적에서 외부 서비스 의존은 발표 당일 오류 위험. 로컬 저장으로 안정성 확보 |
| 지도 | Google Maps SDK | **flutter_map + OpenStreetMap** | Google Maps는 결제 수단 등록 필요. API 키 없는 OSM이 데모에 더 적합 |

## 4. 기말 발표 예상 Q&A

중간 발표 때 실제로 못 들었지만, 기말 발표에서 나올 수 있는 질문들.
특히 "중간 발표 때랑 달라진 게 있지 않나요?"는 반드시 대비해야 함.

---

### Q1. "중간 발표 때 Claude API 쓴다고 했는데, 지금은 Gemini 아닌가요?"

**대응**:
> "맞습니다. 중간 발표 이후 실제 구현하면서 기술적 이유로 Gemini로 바꿨습니다.
> Claude API는 유료 과금 방식이고, Gemini 2.5 Flash는 무료 티어 + Flutter 공식 예제가 잘 정리돼 있어서 발표 데모 환경에 더 적합했습니다.
> AI 엔진을 사용한다는 기획 의도는 동일하고, 프롬프트 설계·JSON 파싱·응답 처리 방식은 모두 실제로 구현했습니다."

---

### Q2. "Firebase 대신 로컬 저장을 쓴 이유가 뭔가요?"

**대응**:
> "의도적인 트레이드오프입니다. Firebase를 연동하면 google-services.json 설정, SHA-1 등록, 웹 플랫폼 추가 설정 등 발표 당일 오류가 날 수 있는 지점이 많습니다.
> SharedPreferences는 외부 서비스 의존 없이 즉시 작동하고 웹·앱 모두 지원합니다.
> 한계(평문 비밀번호 저장)는 알고 있고, 실제 출시 시 Firebase Auth로 교체가 필요하다는 것도 AUTHORING 문서에 명시했습니다."

---

### Q3. "Riverpod 대신 StatefulWidget을 쓴 이유가 뭔가요?"

**대응**:
> "1인 프로젝트, 7개 화면 규모에서 Riverpod는 오버엔지니어링이라고 판단했습니다.
> StatefulWidget으로도 로딩·오류·데이터 상태를 모두 관리할 수 있고, 코드가 더 직관적입니다.
> Riverpod의 AsyncValue가 주는 타입 안전성은 팀 프로젝트·대규모 앱에서 빛나는 장점입니다."

---

### Q4. "날씨가 바뀌었다는 걸 어떻게 감지하나요?"

**대응**:
> "Timer 두 개를 조합했습니다. 앱 시작 후 5초 뒤 즉시 첫 체크, 이후 30분마다 주기 체크입니다.
> 변화 판단은 날씨 조건을 '야외 가능(맑음)'과 '실내 필수(비·폭풍·눈)' 두 그룹으로만 분류해서,
> 그룹 간 전환이 있을 때만 재추천 알림을 보냅니다. 온도 1°C 차이 같은 것은 알림을 보내지 않습니다."

---

### Q5. "AI가 만든 코드를 본인이 이해하고 있나요?"

**대응**:
> "네, 그것이 이 프로젝트의 핵심 원칙입니다.
> AI가 생성한 코드는 반드시 git diff로 확인 후 commit하고, 동작 원리를 직접 설명할 수 있어야만 commit한다는 규칙을 세웠습니다.
> 실제로 5가지 사례에서 AI 초안을 직접 수정했고, 그 이유를 AUTHORING 문서에 기록했습니다. (Gemini 모델 변경, 파싱 로직 추가, 날씨 감지 임계값 단순화 등)"

---

### Q6. "지도는 어떤 라이브러리를 쓴 건가요? Google Maps 아닌가요?"

**대응**:
> "flutter_map 패키지 + OpenStreetMap 타일 서버를 사용했습니다.
> Google Maps SDK는 결제 수단 등록이 필요하고 API 키 관리가 복잡합니다.
> flutter_map + OpenStreetMap은 API 키 없이 동일한 지도 기능(마커, 폴리라인, 줌)을 제공하므로 발표 데모 환경에 더 적합했습니다."

## 5. 기말 발표 Q&A 전략

### 달라진 것들을 먼저 언급한다

발표 시작 전 또는 데모 직전에 "중간 발표 이후 구현 과정에서 일부 기술 스택이 바뀌었습니다"라고 먼저 언급하면 "왜 달라졌나요?" 질문을 자연스럽게 소화할 수 있다.

### "한계를 알고 선택했다"는 프레임

모든 단순화(SharedPreferences, StatefulWidget 등)를 "몰라서 못한 게 아니라 알면서 선택한 트레이드오프"로 설명한다. AUTHORING 문서가 그 근거 자료.

### AI 의존도 질문은 구체적 사례로 답한다

"AI가 만들었다" vs "내가 만들었다" 구도가 아니라, "AI가 초안을 만들고 내가 결정·수정했다"는 협업 프레임을 유지한다. 구체적 수정 사례 5개를 AUTHORING v0.3.0에 기록해뒀으므로 인용 가능.

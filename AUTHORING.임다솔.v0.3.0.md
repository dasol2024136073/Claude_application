## 1. 개요

### AI Agent로 무엇을 했는가

기획 문서 작성부터 코드 생성·리뷰까지 전 과정에 Claude Code(claude-sonnet-4-6)를 AI Agent로 활용했다.
단순히 "질문하고 복붙"하는 방식이 아니라, **Agent에게 맥락을 유지시키고 산출물을 직접 파일로 생성**하게 하는 방식을 택했다.

v0.3.0에서는 OpenWeatherMap 실시간 날씨 연동, flutter_map 지도 시각화, SharedPreferences 경로 저장소, Timer 기반 날씨 모니터링까지 범위가 확장됐다.

### 핵심 원칙

1. AI가 만든 산출물은 반드시 내가 읽고 이해한 후 commit한다.
2. 모호한 결정은 AI가 선택지를 제시하고, 최종 결정은 내가 한다.
3. AI와의 대화 흐름(프롬프트 패턴)을 `notes/` 폴더에 기록해 재사용한다.
4. **(v0.2.0 추가)** AI가 생성한 코드라도 동작 원리를 직접 설명할 수 있어야 commit한다.
5. **(v0.3.0 추가)** 외부 서비스는 "API 키·비용 없는 대안"을 먼저 탐색하고, 그 결정 이유를 AUTHORING에 기록한다.

## 2. Agent 구성

### 사용 도구

| 도구 | 역할 |
|------|------|
| Claude Code (CLI) | 파일 생성·편집, git 조작, 코드 생성 전 과정 |
| 모델 | claude-sonnet-4-6 |
| 실행 환경 | Windows 11, PowerShell / Git Bash |

### Agent에게 부여한 컨텍스트

매 세션 시작 시 Agent는 다음을 자동으로 로드한다.

- `CLAUDE.md` (프로젝트 루트) — 프로젝트 목표, 기술 스택, 금지 사항
- `.planning/*.md` — 기획 문서 전체
- `notes/` — 이전 세션의 LLM Wiki (누적 학습)
- `MEMORY.md` — 프로젝트 현황, 다음 할 일, GitHub URL

이 구조 덕분에 "지난 주에 정한 거 기억해?" 라고 묻지 않아도 Agent가 맥락을 유지한다.

## 3. Skills (slash commands / 반복 작업 자동화)

프로젝트에서 반복적으로 쓴 AI Agent 호출 패턴을 정리한 것.

### /plan — 산출물 먼저 확인 후 실행

```
[작업 설명]. 시작 전에 만들 파일 목록과 각 파일의 핵심 내용을 먼저 보여줘.
내가 OK하면 그때 만들어줘.
```

**왜 씀**: AI가 바로 파일을 만들면 내가 이해 못한 채로 commit하게 됨.
계획을 먼저 보면 "이 부분은 다르게 해줘"라고 수정 지시를 내릴 수 있음.

---

### /explain — 만든 것 설명 요청

```
방금 만든 [파일명]에서 [개념]이 어떻게 동작하는지
비전공자에게 설명하듯 한 단락으로 설명해줘.
```

**왜 씀**: 발표 때 "이게 뭔가요?" 질문에 답하려면 내가 직접 이해해야 함.

---

### /compare — 기술 선택 비교표 요청

```
[기술 A] vs [기술 B]를 이 프로젝트 조건(Flutter, 6주, 1인 개발)에 맞춰
표로 비교해줘. 추천도 한 줄로 해줘.
```

**왜 씀**: ADR 작성 시 대안을 객관적으로 정리하기 위해.

---

### /wiki — LLM Wiki 항목 추가

```
오늘 [주제]에 대해 배운 것을 notes/week{N}-llm-wiki.md에
"## [주제]" 섹션으로 추가해줘. 형식은 기존 항목 스타일 유지.
```

**왜 씀**: 매주 새로 배운 프롬프트 패턴과 실패 사례를 누적 기록하기 위해.

---

### /json-force — LLM에게 구조화된 JSON 강제 (v0.2.0 신규)

```
반드시 아래 JSON 형식으로만 응답해. 마크다운, 설명 텍스트 없이 JSON만:
{ [스키마 예시] }
조건: [필드별 허용값 명시]
```

**왜 씀**: Gemini API처럼 텍스트 생성 모델에게 구조화된 데이터를 요청할 때
형식이 어긋나면 코드에서 파싱 오류가 남. 스키마를 직접 넣어주면 오류율이 크게 줄어듦.

---

### /weather-prompt — 날씨 조건 → LLM 프롬프트 힌트 변환 (v0.3.0 신규)

```
날씨 API 응답에서 조건(condition)을 받아서
LLM에게 전달할 "여행 일정 제약 조건" 문장을 한 줄로 만들어줘.
- 비/폭풍: "실내 전용으로만"
- 맑음: "야외 명소 적극 포함"
- 흐림: "야외/실내 균형"
```

**왜 씀**: 날씨 API 원시 응답("Rain", "Clear" 등)을 그냥 Gemini 프롬프트에 넣으면
LLM이 무시하거나 해석을 임의로 함. 사람이 읽을 수 있는 자연어 지시문으로 변환해야 반영됨.

---

### /monitor-pattern — Timer 기반 폴링 서비스 설계 (v0.3.0 신규)

```
[조건]을 주기적으로 체크하는 서비스 클래스 만들어줘.
- 앱 시작 후 [N]초 뒤 즉시 첫 체크 (데모 시연용)
- 이후 [M]분마다 주기적 체크
- 체크 결과를 콜백(onAlertsUpdated)으로 전달
- static Timer로 관리, stopMonitoring()으로 정리
```

**왜 씀**: "주기적 체크" 요청을 막연하게 하면 AI가 Timer.periodic만 씀.
데모 시연에서는 앱 시작 직후 바로 효과가 보여야 하므로 즉시 첫 체크 + 주기 체크 두 가지를 명시해야 원하는 구조가 나옴.

## 4. Rules (AI와 협업할 때 내가 정한 규칙)

### R-01. AI 출력은 반드시 읽고 commit한다

AI가 파일을 만들면 `git diff`로 확인 후 commit.
"AI가 만들었으니 OK겠지"로 넘기지 않음.

### R-02. 모호한 질문은 먼저 선택지로 좁힌다

"백엔드 뭐 쓸까?" → "Firebase vs Supabase, 이 프로젝트 조건에서 비교해줘"

### R-03. 실패한 프롬프트도 기록한다

잘 안 된 프롬프트 패턴도 `notes/`에 기록. "이렇게 물으면 안 된다"는 것도 암묵지.

### R-04. 한 세션에 한 가지만 요청한다

"기획 문서 만들고, 코드도 짜고, 테스트도 해줘"는 실패함.

### R-05. AI가 모른다고 하면 범위를 좁혀 다시 묻는다

"이 에러 고쳐줘" → "이 에러에서 [특정 부분]만 설명해줘"

### R-06. AI가 생성한 코드에서 직접 수정한 부분을 기록한다 (v0.2.0 신규)

AI가 초안을 만들어도 모델명, 파라미터, 로직의 세부 사항은 내가 직접 검토하고 수정함.
수정 이유를 주석이 아닌 LLM Wiki에 기록해서 다음 세션에 재사용.

### R-07. 외부 서비스는 무료·API 키 불필요 대안을 먼저 탐색한다 (v0.3.0 신규)

지도 기능을 구현할 때 Google Maps SDK를 쓰면 결제 수단 등록 + API 키 관리가 필요.
"flutter_map + OpenStreetMap"은 API 키 없이 동일한 기능을 제공함.
발표 데모 환경에서 API 키 오류로 망가지는 위험을 제거하는 것이 핵심.

### R-08. 저장 시점의 컨텍스트를 메타데이터로 함께 저장한다 (v0.3.0 신규)

경로를 저장할 때 당시의 날씨 조건(`savedWeatherCondition`)을 함께 저장함.
나중에 현재 날씨와 비교해 "재추천 필요 여부"를 판단할 수 있음.
AI에게 "저장 시 날씨도 같이 저장해줘"라고만 하면 스냅샷 설계가 나옴.


## 6. AI가 생성한 코드에서 내가 직접 수정한 사례

### 사례 1: Gemini 모델 변경 (pro → flash)

AI가 처음 생성한 코드는 `gemini-2.5-pro` 사용.
실행해보니 `thinkingBudget: 0` 설정과 충돌해서 API 400 오류.
직접 `gemini-2.5-flash`로 변경 → 정상 작동.

**배운 것**: 모델명은 AI가 제안해도 내가 공식 문서에서 확인하고 검증해야 함.

---

### 사례 2: `_extractJson` 방어 로직 추가 요청

AI가 생성한 초안에는 `jsonDecode(text)` 직접 호출만 있었음.
실제 테스트하니 응답에 마크다운 블록이 포함될 때 파싱 실패 발생.
"코드 블록 제거 후 파싱하는 메서드 분리해줘"라고 재요청 → `_extractJson` 메서드 추가.

**배운 것**: AI 초안은 "이상적 케이스" 기준으로 작성됨. 실제 운영 엣지 케이스는 내가 발견해서 추가 요청해야 함.

---

### 사례 3: SharedPreferences 인증의 보안 한계 인식

AI가 생성한 `AuthService`는 비밀번호를 평문으로 저장함.
"발표 데모용이니 이대로 두되, 실제 출시 시 Firebase Auth로 교체 필요"라는 주석 대신
AUTHORING에 명시적으로 한계를 기록.

**배운 것**: AI가 만든 코드의 보안 한계를 내가 인식하고 문서화하는 것이 책임 있는 AI 활용.

---

### 사례 4: 날씨 변화 감지 임계값 단순화 (v0.3.0 신규)

AI가 처음 생성한 `isSignificantChange()`는 7가지 날씨 조건을 모두 비교하는 복잡한 로직이었음.
"야외 가능 vs 실내 필수"라는 두 카테고리로만 분류하면 충분하다고 판단해 직접 단순화 요청.

```dart
// AI 초안 (복잡):
// 조건별 if-else 7단계 분기

// 내가 수정 요청한 구조:
const outdoor = {'Clear'};
const indoor = {'Rain', 'Drizzle', 'Thunderstorm', 'Snow'};
// outdoor ↔ indoor 전환만 "유의미한 변화"로 판단
```

**배운 것**: AI는 완전한 로직을 만들려는 경향이 있음. 실제 필요한 수준으로 "단순화해줘"라는 명시적 지시가 필요함.

---

### 사례 5: 모니터링 첫 체크 딜레이 30분 → 5초 (v0.3.0 신규)

AI가 처음 만든 `WeatherMonitorService`는 `Timer.periodic(30분)`만 사용.
발표 데모에서 30분을 기다릴 수 없으므로 "앱 시작 후 5초 뒤 즉시 첫 체크"를 추가 요청.

```dart
// 추가 요청 결과:
Timer(const Duration(seconds: 5), () async { ... });   // 즉시 첫 체크
_timer = Timer.periodic(const Duration(minutes: 30), ...); // 이후 주기
```

**배운 것**: "발표 데모 환경"이라는 제약을 명시하면 AI가 시연 친화적인 구조를 제안함.

## 7. 실패한 프롬프트 사례

### 실패 사례 1: "API 연동해줘" — 너무 추상적인 요청

```
// 실패한 프롬프트:
"Gemini API 연동해줘"

// 결과: API 키 관리 방식, 모델명, JSON 스키마, 오류 처리 방식을
//       AI가 임의로 결정해버림. 나중에 여러 곳을 수정해야 했음.

// 개선된 프롬프트:
"GeminiService 클래스 만들어줘.
- 모델: gemini-2.5-flash
- API 키: config.dart의 geminiApiKey 상수 사용
- 입력: destination(String), days(int)
- 출력: TripPlan 객체 (domain/models/trip_plan.dart 기존 모델 사용)
- JSON 응답 파싱 실패 시 Exception throw
- thinkingBudget: 0으로 설정"
```

**교훈**: API 연동 요청은 입력·출력·의존성·예외 처리까지 구체적으로 명시해야 원하는 코드가 나옴.

---

### 실패 사례 2: 테스트를 "자동으로" 만들어달라는 요청

```
// 실패한 프롬프트:
"테스트 만들어줘"

// 결과: 실제 API를 호출하는 테스트, 존재하지 않는 메서드를 호출하는 테스트 등
//       환경에서 실행 불가능한 테스트가 섞여서 나옴.

// 개선된 프롬프트:
"test/unit/ 폴더에 단위 테스트 만들어줘.
- 외부 API 호출 없이 MockTripData만 사용
- 테스트 대상: TripPlan 모델 생성, DayPlan 장소 개수, PlaceItem 필드 검증
- Flutter test 패키지만 사용 (mockito 없이)"
```

**교훈**: 테스트 요청은 테스트 대상, 사용 가능한 의존성, 외부 호출 허용 여부를 명시해야 함.

---

### 실패 사례 3: 날씨 원시 데이터를 그대로 프롬프트에 삽입 (v0.3.0 신규)

```
// 실패한 프롬프트:
"현재 날씨: Rain, 14°C. 이를 반영해서 여행 일정 추천해줘."

// 결과: Gemini가 날씨 언급은 하지만 실내/야외 구성이 바뀌지 않음.
//       "비가 와서 우산 챙기세요" 수준의 텍스트만 추가됨.

// 개선된 프롬프트:
"[날씨 지시] 현재 비가 오는 날씨입니다.
실내 위주 일정으로 구성해줘. 박물관, 실내 시장, 카페 등을 우선 배치하고
야외 장소는 최소화해줘.
[여행 정보] 목적지: 오사카, 기간: 3일"
```

**교훈**: 날씨 조건 그 자체보다 "이것이 일정에 어떤 제약을 주는가"를 명시적인 지시문으로 변환해야 LLM이 반영함.

## 8. 발표 Q&A 대응 전략 (v0.3.0 신규)

기말 발표에서 예상되는 "AI 의존도" 관련 질문과 대응 전략.

### Q1. "이 코드를 AI가 만들었으면 본인이 직접 뭘 한 건가요?"

**대응**: "AI는 초안을 만들고, 저는 이해하고 결정합니다."
- 구체적 근거: AUTHORING 사례 1~5 (모델 변경, 파싱 로직, 보안 한계 기록, 임계값 단순화, 딜레이 조정)
- 핵심 메시지: AI가 만든 코드를 그대로 쓰지 않고 각 결정에 이유가 있음

---

### Q2. "실제 앱에 쓸 수 있나요?"

**대응**: "발표 데모 목적에 최적화된 트레이드오프를 의도적으로 선택했습니다."
- SharedPreferences 인증: 평문 저장 → 출시 시 Firebase Auth로 교체 필요
- 날씨 API: 20개 도시만 매핑 → 추가 도시는 매핑 테이블 확장으로 해결
- 핵심 메시지: 한계를 모르는 게 아니라 알면서 선택한 것

---

### Q3. "Gemini를 쓴 이유가 뭔가요? ChatGPT 아닌가요?"

**대응**: "Google의 Gemini 2.5 Flash를 선택한 기술적 이유가 있습니다."
- 무료 API 티어 제공 (발표 데모 환경에서 비용 부담 없음)
- Flutter와 같은 Google 생태계 — 공식 문서 품질이 높음
- `thinkingBudget: 0`으로 응답 속도 최적화 가능 (Flash 계열 특성)

---

### Q4. "날씨 변화 감지는 어떻게 구현했나요?"

**대응**: "Timer 기반 폴링 + 의미 기반 변화 감지 두 가지를 조합했습니다."
- `Timer(5초)` → 앱 시작 후 즉시 첫 체크 (데모 시연용)
- `Timer.periodic(30분)` → 이후 주기적 체크
- `isSignificantChange()` → "야외 가능" ↔ "실내 필수" 전환만 알림 (온도 1°C 변화는 알림 불필요)

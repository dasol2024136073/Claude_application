## 1. 효과적이었던 프롬프트 패턴

### 패턴 1: 조건을 먼저 정의하고 생성 요청
```
좋은 WBS의 조건을 알려줄테니까 이것을 지켜서 작성해줘.
"명사형으로 산출물을 표현 / 3단계 깊이까지만 / 각 항목은 1~3일 단위로 추정 가능해야 함"
```
**왜 잘 됐나**: AI에게 평가 기준을 미리 주면 기준에 맞게 생성함.  
출력 후 수정 요청 없이 한 번에 원하는 결과가 나왔음.

### 패턴 2: 모호한 부분은 선택형 질문으로 먼저 확인
- 문서 작성 전 AI 추천 방식, 여행 범위, 소셜 기능, 로그인 방식 4가지를 질문으로 먼저 확인함.
- 질문 없이 바로 생성하면 기본 가정으로 작성해버리는 경우가 많음.
- **팁**: "모호한 부분은 나에게 선택형 질문으로 물어본 후 작성해줘" 문구를 요청에 포함하면 AI가 자동으로 확인 먼저 함.

### 패턴 3: 기존 문서를 참조하도록 명시
```
WBS(02-wbs.md)의 항목을 주차별로 배분해서 일정표를 만들어줘
```
**왜 잘 됐나**: AI가 이미 만든 문서를 맥락으로 활용해 WBS ID까지 그대로 연결함.  
문서 간 일관성이 자동으로 유지됨.

## 2. 실패 사례 & 개선

### 실패 1: 자동 push hook 설정 — 한글 경로 인코딩 문제
- **상황**: `.md` 파일 저장 시 자동으로 `git push`되는 hook을 설정하려 했음.
- **문제**: Windows에서 PowerShell 서브프로세스로 스크립트를 실행하면 한글 경로(`학교`, `클로드 기말`)가 깨져서 `git`이 경로를 인식 못함.
- **원인**: PowerShell 파이프라인이 시스템 기본 인코딩(EUC-KR)으로 전달되는데, 서브프로세스는 UTF-8로 읽으려 해서 깨짐.
- **임시 해결**: hook 자동화 대신 직접 `git add → commit → push` 명령어 실행.
- **교훈**: Windows 한글 경로는 자동화 도구에서 자주 문제가 됨. 프로젝트 폴더는 영문 경로 사용 권장.

### 실패 2: git push 충돌
- **상황**: GitHub에서 `00-vision.md`를 직접 수정한 상태에서 로컬에서도 같은 파일을 수정 후 push 시도 → 충돌 발생.
- **해결**: `git stash → git pull → git stash pop → 충돌 해결 → git push` 순서로 처리.
- **교훈**: GitHub 웹에서 직접 파일을 수정하면 로컬과 충돌 가능. 한 곳에서만 수정하거나, 작업 전 항상 `git pull` 먼저.
